Open Precedent

AI-powered legal research platform — design & architecture

Open Precedent is an AI-powered legal research platform that combines hybrid search (pgvector cosine similarity + full-text search with Reciprocal Rank Fusion) with citation-grounded AI synthesis to help legal professionals find and understand case law, statutes, and regulations. The system spans three repositories: a Next.js frontend, a FastAPI backend with agentic AI, and a GPU-accelerated data pipeline.

Hybrid Search. Vector and full-text search execute in parallel via asyncio.gather. Results are merged with Reciprocal Rank Fusion, then grouped by document. This gives the system both semantic understanding (vector) and exact keyword matching (full-text) in a single query.

Pipeline Modules Detail
  • readers.py — Each reader yields RawDocument(source_type, source_id, text, metadata). CSV field size limit raised to 10MB for large opinions.
  • chunker.py — LegalTextChunker uses HuggingFace AutoTokenizer (BGE model). Strips HTML, decodes entities, normalizes whitespace. ~512-token chunks with 64-token overlap.
  • embedder.py — MLXEmbedder (actually sentence-transformers). BGE-large-en-v1.5 in fp16 on MPS/CPU. Thread-safe with threading.Lock. Query prefix: "Represent this sentence for searching relevant passages: "
  • db.py — VectorDB with asyncpg pool (4-20 conns). Batch INSERT + pipeline checkpoint for resume support.
  • delta.py — Incremental updates via CourtListener REST API. Fetches opinions modified in last 24h.
  • alerts.py — Pipeline metrics tracking and Slack notification helpers.

Key Design Decisions
  • pgvector + pg_trgm extensions — Vector similarity (cosine) + trigram text search in a single Postgres instance
  • IVFFlat index (dev) → HNSW index (prod) for vector search. HNSW creation takes 2-6h on 16M rows.
  • Auto-computed tsvector — Trigger trg_chunks_tsvector keeps chunk_tsvector in sync on INSERT/UPDATE
  • Two schema lineages — init.sql creates chunks (dev). Production uses embedding_chunks (pipeline-created). Migrations target production.
  • No ORM — raw SQL throughout, executed via asyncpg conn.fetch/fetchval/fetchrow
  • Health check optimization — Uses reltuples estimate instead of COUNT(*) for instant response on large tables

Moderate resources, production-like config. Monitoring annotations enabled.
Production
HPA auto-scaling, higher memory/CPU limits, OP_DEBUG=false, JSON structured logging.
Pipeline infra. The data pipeline runs on a Mac Studio M4 Max (Apple Silicon). Embedding uses MPS (Metal Performance Shaders) for GPU acceleration with fp16 precision. Optimized for on-premises processing of large legal document datasets before uploading to the Postgres cluster.

Key architectural patterns.
